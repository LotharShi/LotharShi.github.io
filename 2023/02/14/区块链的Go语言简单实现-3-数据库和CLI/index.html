区块链的Go语言简单实现(3):数据库和CLI

Lothar
2023-02-14
Fun › BlockchainGo

简介

到目前为止，我们已经构建了一个拥有PoW机制的区块链系统。我们离一个功能完善的区块链更近了一步，但仍缺少一些关键特征。在今天的部分中，我们会把区块链存入一个数据库中，并构建一个简单的CLI界面来操作区块链。虽然说，区块链的本质是分布式数据库，不过我们暂时会先忽略掉“分布式”而关注“数据库”。

数据库选择

目前在我们的实践中，还咩有数据库。我们每次运行我们的程序时，都会创建区块并且把它们存在内存中。我们并不能重复利用一条区块链，也不能将其共享，因此我们需要把它存在硬盘中。

那么问题来了，该用什么数据库呢？实际上，任何一种都可以。在比特币原始的paper中，并没有提到使用任何一种特定的数据库，所以使用哪一个完全由开发者决定。Satoshi Nakamoto所最早提出的Bitcoin Core中使用的是LevelDB。 而我们即将使用的是…

BoltDB

因为它：

  • 简单且轻量化；
  • 是用Go写的；
  • 并不需要跑服务器；
  • 允许构建我们所需要的数据结构。

在其GitHub上的README里写道：

Bolt is a pure Go key/value store inspired by Howard Chu’s LMDB project. The goal of the project is to provide a simple, fast, and reliable database for projects that don’t require a full database server such as Postgres or MySQL.

Since Bolt is meant to be used as such a low-level piece of functionality, simplicity is key. The API will be small and only focus on getting values and setting values. That’s it.

看起来正合我们的意。让我们再进一步看看。

BoltDB并没有像SQL之类的DBMS的那些table，也没有row或者column。取而代之地，数据以键值对的方式进行存储（就像Golang的maps一样）。而键值对被存储在buckets中，这些buckets被用来对相似的对进行分组（这有些像tables）。因此，如果你想要得到一个value，你需要知道一个bucket和一个key。

关于BoltDB有一点重要的是，在这里并没有数据类型：键和值都是字节数组。既然我们需要储存Go的数据结构（或者说Block）进去，我们需要对其进行序列化，比如写一个能够将Go的数据结构转化为字节流并可以恢复为结构的机制。我们会使用encoding/gob来完成这件事，当然JSON、XML以及Protocal Buffers也可以使用。而之所以选择这个是因为它是标准Go Library的一部分。

数据结构

在开始之前，我们首先要决定我们该如何在DB中进行存储。对此，我们将会参考Bitcoin Core是如何做的。

简单来说呢，Bitcoin Core使用了两个buckets来存储数据。

  1. blocks存储着描述了一条链上所有区块的metadata;
  2. chainstate存储着链的状态，也就是当前所有的未花费的交易输出，和一些metadata。

此外，出于对性能的考虑，区块在硬盘上被分别存储为不同的文件。这样读取每个单独的区块的时候并不需要加载全部区块到内存里。这很好，但为了简单，我们也不会这样做 :)

在区块中，键值对（key -> value）为：

key value
b + 32 字节的 block hash block index record
f + 4 字节的 file number file information record
l + 4 字节的 file number the last block file number used
R + 1 字节的 boolean 是否正在 reindex
F + 1 字节的 flag name length + flag name string 1 byte boolean: various flags that can be on or off
t + 32 字节的 transaction hash transaction index record

在 chainstate，key -> value 为：

key value
c + 32 字节的 transaction hash unspent transaction output record for that transaction
B 32 字节的 block hash: the block hash up to which the database represents the unspent transaction outputs

（具体的解释可以看这里）

既然我们还没有交易，所以暂时只需要有blocks的bucket。而正如前面所说，我们会把整个DB存为一个文件，而不是对每个区块单独存储，所以也不需要和file numbers相关的那些东西。因此，我们的key -> value将为：

  1. 32-byte block-hash -> 序列化了的block结构
  2. l -> 链中最后一块的哈希值

以上就是实现持久化机制所需要了解的全部了。

持久化

我们先从函数NewBlockchain开始。现在的它仅仅创建一个Blockchain的新实例，并且将创世区块加入其中。我们期望它能够完成以下工作：

  1. 打开一个DB文件；
  2. 查看其中是否有区块链存在；
  3. 如果其中有区块链的话：
    1. 创建一个新的Blockchain实例；
    2. 将这个实例的开头设为DB中所存储的最后一个区块对应的哈希值；
  4. 如果其中没有区块链的话：
    1. 创建创世区块；
    2. 存储进DB中；
    3. 将创世区块的哈希值记为最后一个区块的哈希值；
    4. 创建一个新的Blockchain实例，并将其开头的指向设置为创世区块。

检查区块链

现在，所有的区块都被存在了区块链中，所以我们可以再次打开一条区块链冰箱其中添加新的区块。但是在实现了这一切之后，我们缺失了一种特性：我们无法打印我们的区块链因为我们并不再将其存储在任何数组中了。让我们来修复这个问题。

BoltDB允许对一个bucket中的所有键进行迭代，但这些键都是按字节序进行存储的，而我们想要的是区块按照其进入链中的顺序进行打印。此外，因为我们不想把所有的区块都加载进入内存之中，我们会依次读取它们。故而，我们需要一个区块链迭代器：

需要注意的是，一个迭代器最初是指向一条区块链的开头(tip)，因此区块儿们是从头到尾按顺序获取的。实际上，选择一个开头就意味着给一条区块链“投票”。一个区块链可以有多条分支，而其中最长的那条会被认为是主链。在获得一个tip之后（它可以是链上的任何一个区块），我们就可以重新构造整条链，找到它的长度和需要构建它的工作。这同样也意味着，一个 tip 也就是区块链的一种标识符。

BlockchainIterator 只会做一件事情：返回链中的下一个区块。
